# Architecture

The corpus infrastructure is four moving parts and three on-disk artifacts. Every piece exists to make worldbuilding content searchable, navigable, and editable by Claude without manual file shuffling.

## The four components

1. **Docker filesystem MCP** — The standard Anthropic filesystem server, run in Docker, with `/corpus` bind-mounted to `D:\claude\filesystem\` on the host. Provides `filesystem:read_text_file`, `write_file`, `edit_file`, `list_directory`, `search_files`, etc. This is the only way Claude reads or writes corpus files.
2. **Index builder** — `Python/build_indexes.py`, driven by `Python/indexer.cfg` via `Python/cfg_loader.py`. A single `os.walk` over the corpus produces all three derived artifacts in one pass.
3. **Corpus-search MCP server** — `Python/search_mcp_server.py`. Custom Python MCP server. Opens `index/search_index.db` read-only, exposes `search_corpus` and `index_status` tools to Claude.
4. **Index-tools MCP server** — `Python/index_tools_mcp_server.py`. Custom Python MCP server. Wraps `build_indexes.py` as a subprocess, exposes `rebuild_indexes(load=...)` to Claude.

## The three artifacts (all in `/corpus/index/`)

1. **`directory_index.md`** — YAML header + compressed directory tree (dirs only). Read at session start so Claude knows the corpus shape without making dozens of `list_directory` calls.
2. **`directory_index_with_files.md`** — Same shape but includes filenames under each directory. Loaded on demand for filesystem-heavy work.
3. **`search_index.db`** — SQLite database with two tables (`corpus_fts` FTS5 virtual table + `corpus_meta` companion table). Powers `search_corpus`.

All three are gitignored (the entire `/index/` folder is gitignored) and regenerated by `build_indexes.py`. They are derived state — never edit them directly.

## Data flow

```
[USER edits a corpus file]
    │
    ▼
[Docker filesystem MCP]  ──── /corpus mount ────  D:\claude\filesystem\
    │
    │  (file change happens; nothing auto-rebuilds)
    │
    ▼
[Claude or user triggers rebuild]
    │
    ├──► via MCP tool:  index-tools:rebuild_indexes
    │       │
    │       └─► subprocess: python build_indexes.py --no-pause
    │
    └──► via Explorer:  double-click Python/refresh_indexes.bat
            │
            └─► python build_indexes.py
    │
    ▼
[build_indexes.py]
    │  reads indexer.cfg via cfg_loader.py
    │  single os.walk over /corpus
    │  writes three outputs to /corpus/index/:
    │    ├─► directory_index.md
    │    ├─► directory_index_with_files.md
    │    └─► search_index.db   (corpus_fts + corpus_meta tables)
    │
    ▼
[corpus-search MCP server]
    │  opens search_index.db read-only on each query
    │  serves search_corpus + index_status to Claude
    │
    ▼
[CLAUDE]
    ├──► reads directory_index.md at session start
    ├──► calls search_corpus(query, filters...) for content search
    └──► calls rebuild_indexes(load=...) when structure drifts
```

## Where state lives

| State                              | Location                                              | Editable? |
|------------------------------------|-------------------------------------------------------|-----------|
| Source content (worldbuilding)     | `/corpus/World_Building/`, `/corpus/Core_Rules/`, ... | Yes       |
| Project rules                      | `/corpus/file_system_instructions.md`                 | Yes       |
| Indexer configuration              | `/corpus/Python/indexer.cfg`                          | Yes       |
| Index builder source               | `/corpus/Python/build_indexes.py`                     | Audit before edits (see `Security_Audit.md`) |
| MCP server sources                 | `/corpus/Python/*_mcp_server.py`                      | Audit before edits |
| Directory indexes                  | `/corpus/index/*.md`                                  | No (derived) |
| Search database                    | `/corpus/index/search_index.db`                       | No (derived) |
| MCP server registration            | `claude_desktop_config.json` (outside corpus)         | Yes (restart required) |

## What requires what to restart

- Edit `indexer.cfg` or `build_indexes.py` → run `rebuild_indexes` or the bat file. No restart needed.
- Edit `search_mcp_server.py` or `index_tools_mcp_server.py` → **restart Claude Desktop**. The servers are long-running stdio subprocesses and don't pick up source changes.
- Edit the Dockerfile or `docker-compose.yml` → rebuild the container (see `Docker_Filesystem.md`).

## Why this design

- **SQLite + FTS5** for search because: zero setup, single-file portability, built-in ranking, no daemon. The whole DB ships in one binary file.
- **Two tables, not one**, because FTS5's tokenizer splits on hyphens and underscores. Values like `setting-document` aren't searchable as FTS5 tokens, so `corpus_meta` holds them for SQL equality lookups. See `Search_Server.md`.
- **Cfg-driven, not hardcoded**, because exclusion rules drift faster than the script. `Trash/` and `Sheet_Import/` policies have changed multiple times; the cfg made each change a one-line edit.
- **Docker for the filesystem MCP** because the corpus is the only thing it needs to see, and `/corpus` mounted at a fixed path means Claude's mental model of paths matches the container regardless of host OS.
- **Subprocess rebuild, not in-process**, because the MCP server stays cheap and stable — the heavyweight code lives in a script that exits cleanly each run.
